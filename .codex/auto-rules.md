# 自动化规则
# 控制系统自进化、会话初始化等自动化行为

## 规则 1：会话启动逻辑

**每次新会话启动时**，主 Agent 必须执行以下自动化序列：

1. **项目状态巡检**：
   - 读取 `.codex/state.json` 获取上次会话的进度状态
   - 扫描 `docs/` 目录确认实际存在的文档
   - 对比状态文件和实际情况，如有不一致以实际情况为准并更新 state.json
   - 输出当前阶段摘要

2. **信号进化检查**：
   - 读取 `.codex/signals/signals.json`
   - 统计未处理信号数量
   - 如有未处理信号 >= 1，触发进化流程：
     a. 提示用户有待处理信号
     b. 询问是否立即执行进化（或稍后）
     c. 如同意，派遣 Evolution Runner 子 Agent
     d. 展示进化建议，等待用户确认
     e. 确认后更新对应规则文件
     f. 在 signals.json 中将相关信号标记为 processed = true

3. **文档一致性检查**：
   - 检查 code 目录的最后修改时间与 CHANGELOG 的最后记录是否匹配
   - 如有 code 变更但 CHANGELOG 未更新，提示用户
   - 如有 PRD/DesignBrief/Plan 更新但下游未同步，标记差异

## 规则 2：信号消化流程

**信号消化是 Evolution Runner 的核心工作流程**：

1. **触发条件**：
   - 新会话启动时发现有未处理信号（自动触发）
   - 用户手动触发进化（通过 AutomationManager 或直接请求）

2. **处理步骤**：
   - 读取所有未处理信号
   - 按 source_skill 分组
   - 每组内按 type 归类，合并同类信号
   - 对每组生成规则变更建议
   - 每个建议标注变更类型

3. **向用户确认的格式**：

```
🔍 检测到 [N] 个待处理用户反馈信号

[信号列表]
- #[ID] [类型]：[摘要]（来源：[Skill名]）

建议变更：
1. [新增/替换/删除] [目标文件]：[变更摘要]
2. ...

确认执行以上变更？(Y/N)
```

4. **用户确认后的操作**：
   - 执行规则文件变更
   - 将信号标记为 processed = true
   - 在 CHANGELOG 中记录规则进化

## 规则 3：规则生命周期管理

**新增规则入库逻辑**：
- 新增规则优先写入对应的 `.agents/` Skill 文件
- 全局性规则写入 `agents.md`
- 程序可执行的门禁类规则写入 `.codex/hooks/`
- 子 Agent 配置类规则写入 `.codex/subagents/`

**冗余规则检测与删除**：
- 每 30 天或每 50 条信号处理后，执行规则健康检查
- 检查标准：
  - 是否有超过 90 天未被任何上下文使用的规则
  - 是否有被新规则完全覆盖的旧规则
  - 是否有与其他规则冲突的规则
- 检测到冗余规则后，生成删除建议
- 用户确认后方可删除

**规则版本标记**：
- 每次规则变更后，在文件中增加元数据标记：
  ```
  <!-- last-evolved: 2026-07-02 -->
  <!-- evolution-id: EVO-20260702-001 -->
  ```

## 规则 4：进化边界

**进化允许的范围**（仅以下内容）：
- 问题库：追加新的逼问式问题、扩展追问维度
- 执行标准：优化验收标准的具体描述
- 禁令：补充新的禁令规则
- 好/坏标准：明确什么产出合格、什么必须打回
- 审查清单：增加新的审查维度

**进化不允许的范围**：
- 核心流程架构（六个环节的定义和顺序）
- Skill 的触发条件逻辑（何时调用什么 Skill）
- 子 Agent 派发纪律（主 Agent 负责制原则）
- Go 指令执行规范（三要素原则）
- hooks 的门禁拦截逻辑
