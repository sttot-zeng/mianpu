# Stage Gate Hooks
# 节点门禁规则 —— 硬卡流程节点，不满足条件禁止进入下一阶段
# 优先于所有自然语言规则执行

## 门禁 1：需求文档未完成不得启动设计
- **检查条件**：当用户请求进入设计阶段时
- **检查逻辑**：
  - 检查 `docs/` 目录下是否存在 `PRD-*.md` 文件
  - 检查 PRD 文件是否包含所有必需章节（产品上下文、开发范围、用户故事、核心流程、功能清单、技术依赖）
  - 检查 PRD 文件最后是否包含用户确认标记
- **拦截行为**：
  - 如不满足条件：拒绝进入设计阶段，返回缺失项清单，提示继续完善需求
  - 如满足条件：允许进入，记录同步 `state.current_stage = "design"`

## 门禁 2：设计文档未完成不得启动编码
- **检查条件**：当用户请求或 AI 判断应进入开发计划阶段时
- **检查逻辑**：
  - 检查 `DesignBrief-*.md` 是否存在（如跳过 DesignMaker）
  - 或检查 `PRD-*.md` 是否存在（跳过设计环节时）
  - 检查文档是否包含设计原则、页面清单等核心内容
  - 检查文档是否有用户确认标记
- **拦截行为**：
  - 如不满足条件：拒绝进入开发阶段，返回缺失项清单
  - 如满足条件：允许进入开发计划阶段

## 门禁 3：开发计划未确认不得开始编码
- **检查条件**：当要开始编写实际代码时
- **检查逻辑**：
  - 检查 `Plan-*.md` 是否存在
  - 检查每个 Phase 是否定义了验收标准
  - 检查技术选型、依赖关系是否明确
  - 检查是否有用户确认标记
- **拦截行为**：
  - 如不满足条件：拒绝开始编码，要求先完成开发计划
  - 如满足条件：记录 `state.current_phase = "phase-1"`

## 门禁 4：代码未通过审查不得进入下一阶段
- **检查条件**：一个开发任务完成后，准备开始下一个任务时
- **检查逻辑**：
  - 检查最近一次代码审查是否通过
  - 检查是否所有严重/重要问题已修复并重新审查通过
  - 如有并行任务：检查所有并行任务是否均已通过审查
- **拦截行为**：
  - 如有未关闭的审查问题：阻止进入下一任务，要求先修复问题
  - 如审查通过：允许继续

## 门禁 5：未通过隐私审计不得发布
- **检查条件**：进入发布阶段时
- **检查逻辑**：
  - 检查是否已执行隐私审计流程
  - 检查审计报告是否有未解决的红旗标记
- **拦截行为**：
  - 如未执行审计：阻止发布，要求先完成隐私审计
  - 如审计有未解决问题：阻止发布，要求先修复

## 门禁 6：Go 指令必须包含三要素
- **检查条件**：收到或生成 Go 指令时
- **检查逻辑**：
  - 检查是否包含明确目标
  - 检查是否包含可量化的完成标准
  - 检查是否包含可执行的验收方式
- **拦截行为**：
  - 如缺少任一要素：拒绝执行，要求补充完整
  - 如三要素齐全：允许执行

## 通用门禁行为
- 每次门禁触发必须输出清晰的拦截原因
- 必须给出具体的解决方案建议
- 用户可通过显式指令绕过门禁，但需记录在状态文件中
