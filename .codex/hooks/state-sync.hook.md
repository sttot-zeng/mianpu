# State Sync Hooks
# 状态同步规则 —— 代码/文档变更时自动触发关联文档的同步更新

## 同步规则 1：代码变更 -> 更新 CHANGELOG
- **触发条件**：代码文件发生增删改（通过 git diff 检测）
- **执行逻辑**：
  - 如果 `docs/CHANGELOG.md` 不存在则创建
  - 按日期追加变更记录
  - 变更格式：`YYYY-MM-DD：[修改类型] [影响范围] - [变更摘要]`
  - 修改类型：新增 / 修改 / 修复 / 删除 / 重构
- **例外**：.md 文档本身的变更不触发 CHANGELOG 更新

## 同步规则 2：需求变更 -> 通知后续环节
- **触发条件**：PRD 文档发生修改
- **执行逻辑**：
  - 检查变更内容是否影响开发范围和功能清单
  - 如有影响，在项目状态文件中记录变更通知
  - 在 `agents.md` 的状态巡检结果中显示"需求已变更，建议重新评估"
  - 不自动修改已有代码或计划，但提示用户影响

## 同步规则 3：开发计划变更 -> 更新任务状态
- **触发条件**：Plan 文档发生修改（阶段增减、依赖关系变更）
- **执行逻辑**：
  - 检查是否有已完成的 Phase 被标记删除或修改
  - 如有：记录冲突，提示用户确认已完成工作如何处理
  - 如无冲突：更新任务状态记录

## 同步规则 4：状态文件同步
- **项目状态文件**：`.codex/state.json`
- **维护规则**：
  - 每次完成一个环节后更新 `current_stage`
  - 每次完成一个 Phase 后更新 `completed_phases`
  - 每次信号捕获后更新 `pending_signals`
  - 状态文件格式：

```json
{
  "project_name": "",
  "current_stage": "requirement",
  "completed_stages": [],
  "current_phase": null,
  "completed_phases": [],
  "last_review_status": null,
  "pending_signals": 0,
  "last_updated": "2026-07-02T16:00:00+08:00"
}
```

## 同步规则 5：跨会话状态持久化
- 所有同步结果必须持久化到文件系统
- 新会话启动时先读取状态文件恢复上下文
- 状态文件变更前先读取最新版本，避免覆盖其他会话的更新
