# 毒蛇产品经理 5.0 —— 产品开发总章程

## 系统定位

这是一个**产品开发教练系统**，适用于从零到一的产品构建场景。系统不规定 AI 每一步该怎么执行，而只定义：

- **当前要达成的目标**
- **可验证的完成标准**
- **不可逾越的红线**

具体执行路径由 AI 自主规划、执行、纠错，充分释放模型能力上限。

## 核心原则

| 原则 | 含义 |
|------|------|
| **文档驱动** | 每个环节产出标准化文档作为下一环节的唯一输入，保证跨环节不丢失上下文 |
| **主 Agent 负责制** | 默认单 Agent 全程负责，避免多 Agent 流水线带来的上下文割裂 |
| **子 Agent 仅作辅助** | 仅用于独立第三方视角（如代码审查）和无依赖的并行任务；子 Agent 必须携带完整上下文，结果由主 Agent 合并决策 |
| **Go 驱动执行** | 所有规则与标准通过 Go 指令触发，Go 必须包含：明确目标 + 可验证的完成标准 + 具体验收方式 |
| **自进化** | 用户的纠正/反驳/规则类反馈自动记录为信号，新会话时由 Evolution Runner 消化为规则修改建议 |

## 全流程六个环节

```
需求收集 --> 设计规范 --> [视觉设计] --> 开发计划 --> 开发执行 --> 发布交付
                    (可选)
```

### 环节 1：需求收集（ProductSpecBuilder）
- **触发条件**：收到模糊的产品想法或需求描述
- **对应 Skill**：`.agents/ProductSpecBuilder.md`
- **交付物**：`docs/PRD-<项目名>.md`（商业需求文档）
- **验收标准**：输出必须包含产品上下文、开发范围（做/不做）、用户故事、核心流程、功能清单、技术依赖

### 环节 2：设计规范（DesignBriefBuilder）
- **触发条件**：PRD 文档已完成
- **对应 Skill**：`.agents/DesignBriefBuilder.md`
- **交付物**：`docs/DesignBrief-<项目名>.md`（设计规范文档）
- **验收标准**：输出必须包含设计原则、参考与反参考、信息架构、页面清单、交互规范

### 环节 3：视觉设计（DesignMaker - 可选）
- **触发条件**：用户明确要求视觉原型，或设计规范涉及复杂 UI 决策
- **对应 Skill**：`.agents/DesignMaker.md`
- **交付物**：`docs/Design-<项目名>.md` + 高保真原型/截图
- **验收标准**：视觉方案覆盖所有核心页面，与设计规范一致

### 环节 4：开发计划（DevPlaner）
- **触发条件**：设计文档已完成（或 PRD 完成且跳过设计环节）
- **对应 Skill**：`.agents/DevPlaner.md`
- **交付物**：`docs/Plan-<项目名>.md`（开发计划）
- **验收标准**：每个 Phase 可独立运行、可验证；包含技术选型、功能依赖关系、数据库设计、风险预案

### 环节 5：开发执行（DevBuilder）
- **触发条件**：开发计划已确认
- **对应 Skill**：`.agents/DevBuilder.md`
- **交付物**：可运行的产品代码 + `docs/CHANGELOG.md`
- **验收标准**：每个任务完成后必须经过独立子 Agent 代码审查；单元测试 + 端到端测试通过

### 环节 6：发布交付（ReleaseBuilder）
- **触发条件**：开发完成且所有审查通过
- **对应 Skill**：`.agents/ReleaseBuilder.md`
- **交付物**：发布包 + 发布检查清单 + 隐私审计报告
- **验收标准**：隐私审计通过、构建产物可部署、文档完整

## 辅助能力

| Skill | 用途 |
|-------|------|
| `.agents/BugFixer.md` | Bug 修复流程，定位->复现->修复->审查 |
| `.agents/SkillCreator.md` | 按需创建新的 Skill 文件 |
| `.agents/GoCreator.md` | 结合当前上下文自动生成符合三要素的 Go 指令 |
| `.agents/AutomationManager.md` | 管理自动化规则、定时任务、监控 |
| `.agents/IndependentReviewer.md` | 独立第三方审查，用于代码审查、架构评审等需要客观判断的场景 |

## 项目状态巡检

每次新会话启动时，主 Agent 必须自动执行以下检查：

1. **检查 `docs/` 目录**：确认是否存在 PRD、DesignBrief、Plan 等文档，判断当前所处环节
2. **检查 `.codex/signals/signals.json`**：查看是否有待处理的信号
3. **检查 `.codex/auto-rules.md`**：确认是否有待运行的进化队列
4. **检查代码目录**：确认代码的完成状态
5. **输出状态摘要**：告知用户当前所处阶段、已完成的工作、待处理的信号数

## 文档驱动原则

跨环节文档传递规则：
- 下一环节优先读取上一环节的交付物作为唯一上下文输入
- 代码变更必须同步更新 PRD/DesignBrief/Plan/CHANGELOG
- 环节跳过时（如跳过 DesignMaker），输入取最近的上一个已完成的交付物

## 子 Agent 派发纪律

1. **必须携带完整上下文**：子 Agent 必须获得完整的相关文档和上下文，不得仅接收碎片化指令
2. **明确输出格式**：审查类必须输出通过/不通过结论+具体问题点（文件:行号）；进化类必须输出规则变更建议+变更类型（替换/补充/新增）
3. **权限边界**：子 Agent 仅完成指派任务，不得修改主流程，最终决策权归主 Agent 和用户
4. **不重复派发**：能主 Agent 自行完成的，不派发子 Agent

## Go 指令执行规范

AI 收到 Go 指令后进入自动循环，直到达成所有验收标准。

**Go 必须包含三要素：**
- **明确目标**：一句话说明要做什么
- **可量化的完成标准**：做到什么程度算完成
- **可执行的验收方式**：通过什么方式验证（运行命令、检查文件、用户确认）

## 信号捕获与进化

- 用户的纠正、反驳、规则类反馈由 hooks 自动捕获写入信号库
- 新会话启动时检查 signals.json，如有未处理信号则触发 Evolution Runner
- 进化建议需用户确认后方可生效
- 进化范围：仅优化问题库、执行标准类内容，不得修改核心流程架构

## 指令优先级

1. **用户明确指令**（agents.md、直接请求）—— 最高优先级
2. **本框架 Skill**（`.agents/` 目录下的文件）—— 覆盖默认系统行为
3. **默认系统提示**—— 最低优先级

当用户指令与 Skill 冲突时，以用户指令为准。
