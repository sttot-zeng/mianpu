# DesignBriefBuilder —— 设计规范

## 适用场景

- PRD 已完成，需要确定产品视觉风格和交互规范
- 用户提出"高级感""简洁""大气"等模糊设计诉求时
- 产品进入 UI 设计阶段前

## 宏观骨架

**先打地基**：确定参考方向与反参考、整体视觉气质、信息密度
**再抠细节**：逐层确定色彩体系、字体规范、组件风格、交互模式

## 验收标准

1. **交付物**：`docs/DesignBrief-<项目名>.md`，必须包含：
   - **设计原则**：2-5 条核心设计原则（如"内容为王""操作直觉化"）
   - **参考与反参考**：参考风格列表 + 明确说明哪些风格不想要
   - **信息架构**：页面导航结构、内容组织层级
   - **页面清单**：所有需要设计的页面/视图列表
   - **交互规范**：关键交互模式（点击/滑动/拖拽/弹窗方式）
   - **状态覆盖**：空状态、错误状态、加载状态的视觉策略

2. **验证方式**：
   - 用户逐节确认每个设计决策
   - 每个设计维度都有明确的备选项而非开放式问题
   - 参考风格有实际案例（截图/链接）

## 红线禁令

- **禁止用模糊词替代设计决策**："好看""高级""现代"必须拆解为具体设计特征
- **禁止不做选择**：每个维度必须推进到具体选择，不遗留"都行"
- **禁止忽视移动端**：如产品需要移动端适配，必须明确响应式策略
- **禁止设计过度**：不追求完美视觉，只追求足够清晰的指导规范

## 设计维度问题库

每个维度提供备选项引导用户选择：

### 参考方向
- 能否提供 1-3 个参考产品/网站？
- 喜欢参考产品的哪些方面？（整体气质 / 色彩 / 布局 / 交互细节）
- **合格**：用户能指出具体的参考和喜欢的点
- **不合格**："像某某产品"但没有具体说明喜欢什么

### 视觉气质
- [A] 专业商务型：稳重、克制、信任感
- [B] 创意创新型：个性、活力、与众不同
- [C] 简洁干净型：留白多、内容突出、干扰少
- [D] 数据驱动型：图表丰富、信息密集、效率优先

### 主题深浅
- [A] 浅色主题（白色/浅灰背景，适合内容型产品）
- [B] 深色主题（深色背景，适合创意/观看型产品）
- [C] 双主题（同时支持浅色和深色模式）

### 信息密度
- [A] 极简型：大量留白，每次只展示核心内容
- [B] 均衡型：适当的卡片和列表组合
- [C] 密集型：表格/列表为主，信息密度高

### 色彩体系
- 主色调偏好？（品牌色/行业色/用户倾向色）
- 是否需要多色体系？（辅助色/语义色/数据色）
- 色彩使用原则？（大面积用色 / 点缀用色 / 数据用色）

### 字体规范
- [A] 系统原生字体（加载快，无版权问题）
- [B] 自定义字体（有品牌调性，需考虑加载和版权）
- 标题字体风格偏好？

### 圆角使用
- [A] 直角（专业感、效率感）
- [B] 小圆角 4-6px（温和但有边界感）
- [C] 大圆角 8-16px（亲和、友好的品牌调性）
- [D] 极圆角（极其圆润、年轻化）

### 导航样式
- [A] 侧边栏导航（信息层级深、功能多）
- [B] 顶部导航（信息扁平、内容为王）
- [C] 底部 Tab 导航（移动端常见）
- [D] 混合导航（侧边栏 + 顶部面包屑）

### 需要规避的反参考风格
列出明确的"我们不这样做"的风格：
- SaaS 仪表盘风格？（可能太办公化）
- 电商促销风格？（可能太商业）
- 社交媒体风格？（可能太娱乐）
- 具体需要避免的产品名称？
