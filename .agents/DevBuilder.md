# DevBuilder —— 开发执行

## 适用场景

- 开发计划（Plan）已确认
- 需要开始或继续编写产品代码

## 宏观骨架

**先打地基**：确认项目环境、读取 Plan 当前 Phase、同步状态
**再抠细节**：按 Phase 逐个任务执行，每个任务后走开发-审查闭环

## 验收标准

1. **交付物**：
   - 可运行的产品代码
   - `docs/CHANGELOG.md`（每次代码变更后更新）
   - 每个任务对应的单元测试和端到端测试

2. **验证方式**：
   - 每个任务完成后必须通过独立子 Agent 代码审查
   - 所有单元测试通过
   - 端到端核心链路验证通过
   - 代码无已知严重/重要审查问题未修复

## 红线禁令

- **禁止跳过代码审查**：每个任务完成后必须派发独立审查子 Agent
- **禁止未通过审查进入下一任务**：严重/重要问题必须修复并重新审查通过
- **禁止先写代码后补测试**：测试必须与代码同期编写（TDD 优先）
- **禁止乱改 Plan**：开发中发现需要变更计划时，先更新 Plan 文档，再执行
- **禁止不更新 CHANGELOG**：每次代码变更必须同步更新变更日志

## 开发-审查闭环流程

```
┌─────────────────────────────────────────────────────────────┐
│  1. 读取 Plan，确认当前要完成的任务                           │
│  2. 实现任务（编码 + 测试）                                  │
│  3. 自检：确保所有测试通过                                    │
│  4. 派发 Spec Reviewer 子 Agent 审查规格合规                  │
│  5. 如不合规 -> 返工修复 -> 回到步骤 4                       │
│  6. 派发 Code Reviewer 子 Agent 审查代码质量                  │
│  7. 如有问题 -> 返工修复 -> 回到步骤 6                       │
│  8. 审查通过 -> 更新 CHANGELOG                               │
│  9. 标记 Task 完成，进入下一 Task                             │
└─────────────────────────────────────────────────────────────┘
```

## 子 Agent 派发明细

### Spec Reviewer 规格审查
- 位置：`.codex/subagents/spec-reviewer.yaml`
- 需要提供的上下文：原始需求描述 + 实现者报告 + 工作目录
- 期望输出：合规/不合规结论 + 遗漏/多余/误解清单

### Code Reviewer 代码审查
- 位置：`.codex/subagents/code-reviewer.yaml`
- 需要提供的上下文：开发任务描述 + 变更代码范围 + 审查标准
- 期望输出：通过/不通过结论 + 问题清单（严重/重要/建议）

## 代码变更同步规则
- 代码变更涉及需求变化时，更新 PRD 中的功能清单
- 代码变更影响设计决策时，在设计规范中添加备注
- 代码变更影响阶段划分时，更新开发计划
- 所有变更记录到 CHANGELOG

## 测试要求
- 关键业务逻辑必须有单元测试（覆盖率 >= 70%）
- 核心流程必须有端到端测试
- 边界情况和异常流程必须有测试覆盖
- 测试必须是可重复的（不依赖外部环境状态）
