# BugFixer —— Bug 修复

## 适用场景

- 用户报告产品运行异常或行为不符合预期
- 测试过程中发现缺陷

## 宏观骨架

**先定位**：确认 Bug 表现、复现条件、影响范围
**再修复**：最小化修复、补充测试、代码审查

## 验收标准

1. **完整流程**：定位 -> 复现 -> 修复 -> 测试 -> 审查
2. **交付物**：
   - Bug 修复代码
   - 回归测试（防止再次出现）
   - CHANGELOG 中的修复记录
3. **验证方式**：
   - Bug 原始症状不再出现
   - 回归测试通过
   - 代码审查通过

## 红线禁令

- **禁止未定位原因就尝试修复**：必须先找到根因
- **禁止只修症状不修根因**：必须解决根本问题
- **禁止不补测试**：修复必须附带验证该 Bug 的测试
- **禁止大范围重构式修复**：最小化改动原则

## Bug 修复流程

```
1. 确认 Bug 表现：用户在什么场景下遇到了什么问题？
2. 确定复现步骤：如何稳定复现？
3. 定位根因：检查日志、代码走读、二分定位
4. 评估影响：影响哪些用户？影响哪些功能？
5. 设计修复方案：最小改动、不引入副作用
6. 实施修复：编码
7. 编写回归测试：确保 Bug 不再出现
8. 验证：原始症状消失 + 回归测试通过 + 全量测试通过
9. 代码审查：派发 Code Reviewer 子 Agent
10. 记录：更新 CHANGELOG、关闭 Issue（如有）
```

## Bug 严重程度分级

| 级别 | 定义 | 处理要求 |
|------|------|---------|
| **P0 崩溃** | 核心功能不可用、数据丢失、安全漏洞 | 立即修复，不可进入下一阶段 |
| **P1 严重** | 主要功能异常、影响用户体验但可绕行 | 本周内修复 |
| **P2 一般** | 非核心功能异常、UI 异常 | 列入待办 |
| **P3 轻微** | 文案错误、样式偏差 | 可延后 |

## Bug 记录格式

```
### Bug #[ID]：[标题]
- **严重级别**：P0/P1/P2/P3
- **发现时间**：YYYY-MM-DD
- **环境**：浏览器/设备/版本
- **复现步骤**：1. 2. 3.
- **实际结果**：[Bug 表现]
- **预期结果**：[应该是什么]
- **根因**：[代码中的根本原因]
- **修复方案**：[如何修复]
- **状态**：已识别 / 修复中 / 已修复 / 已验证
```
