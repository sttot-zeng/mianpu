# DevPlaner —— 开发计划

## 适用场景

- PRD（或 DesignBrief）已完成并确认
- 需要将需求拆分为可执行的开发阶段

## 宏观骨架

**先打地基**：确定技术栈、整体架构、数据模型
**再抠细节**：按 Phase 拆分任务、定义依赖关系、明确验收标准

## 验收标准

1. **交付物**：`docs/Plan-<项目名>.md`，必须包含：
   - **技术选型**：前端/后端/数据库/部署方案，含选型理由
   - **架构概览**：系统模块图或组件关系说明
   - **功能依赖关系**：任务之间的前后置依赖图
   - **Phase 分解**：每个 Phase 可独立运行、可验证效果
     - Phase 0（基础设施）：项目骨架、CI/CD、数据库初始化
     - Phase 1（核心链路）：最小可用版本的核心流程
     - Phase 2（功能完善）：P1 功能
     - Phase 3（打磨优化）：P2 功能、性能优化、Bug 修复
   - **数据库设计**：核心数据模型和关系
   - **开发规则**：代码规范、命名约定、测试要求
   - **风险预案**：主要技术风险的应对策略

2. **验证方式**：
   - 用户逐 Phase 审阅确认
   - 每个 Phase 的验收标准可量化（"搜索功能支持关键词模糊匹配"而非"搜索好用"）
   - 所有 Phase 顺序执行完成后能产出完整功能

## 红线禁令

- **禁止长期无可见产出的 Phase**：每个 Phase 必须在 1-2 次对话内可验证
- **禁止忽略依赖关系**：如果 Phase 2 依赖 Phase 1，必须明确标注
- **禁止技术选型不解释**：每个选型决策必须附带理由
- **禁止代码复用遗漏**：如果已有原型代码，必须明确复用哪些部分

## Phase 分解原则

### Phase 0：基础设施
- 项目脚手架搭建
- 数据库初始化（Schema、迁移脚本）
- 基础架构组件（路由、状态管理、HTTP 客户端）
- 测试框架配置
- **验收**：项目可启动运行，数据库连接正常

### Phase 1：核心链路
- 实现用户核心目标的完整流程
- 仅包含 P0 功能，不处理边界情况
- **验收**：用户可以通过完整流程达成核心目标

### Phase 2：功能完善
- P1 功能实现
- 边界情况处理
- 错误提示和异常流程
- **验收**：所有 P0+P1 功能可用

### Phase 3：打磨优化
- P2 功能实现
- 性能优化
- 视觉细节打磨
- 已知 Bug 修复
- **验收**：全功能可用，无已知严重问题
