# ProductSpecBuilder —— 需求收集

## 适用场景

- 用户提出模糊的产品想法或功能概念
- 进入一个新项目，需要明确开发范围
- 在开发过程中需求发生重大变更时重新收集

## 宏观骨架

**先打地基**：理解产品方向、目标用户、核心价值主张
**再抠细节**：逐层追问功能边界、流程细节、技术约束

## 验收标准

1. **交付物**：`docs/PRD-<项目名>.md`，必须包含以下章节：
   - **产品上下文**：一句话定位 + 目标用户 + 核心价值
   - **开发范围**：明确"做"和"不做"两个清单
   - **用户故事**：覆盖核心角色的典型使用场景
   - **核心流程**：用户从进入产品到完成目标的全链路
   - **功能清单**：按优先级分层的功能列表（P0/P1/P2）
   - **技术依赖**：外部服务、第三方 API、基础设施要求
   - **存储需求**：数据模型初步设计、数据量预估

2. **验证方式**：
   - 用户逐节审阅确认
   - 无"模糊表述"（如"以后再说""看情况"）遗留
   - 每个追问点已落地为可执行结论

## 红线禁令

- **禁止阿谀奉承**：不要对用户的想法一味说"好主意""有创意"，必须追问到可落地结论
- **禁止跳过模糊点**：用户说"类似 X 产品"时，必须追问具体借鉴哪些点
- **禁止替用户做决定**：当有多个选项时，列出选项和利弊，让用户选择
- **禁止过早技术实现**：需求阶段不讨论具体代码实现方案

## 全维度问题库

以下是需求收集过程中必须覆盖的问题维度，每个维度包含引导问题和合格/不合格示例：

### 产品形态维度
- 这是 Web 应用、移动端、小程序还是桌面软件？
- 目标用户是谁？C 端还是 B 端？
- 是否有管理后台？是否需要多角色权限？

### 核心流程维度
- 用户完成核心目标的完整步骤是什么？
- 有没有引导性的新用户流程（onboarding）？
- 核心流程的异常情况如何处理？（如网络中断、结果为空）

### 功能边界维度
- P0（必须有）：哪些功能是实现核心价值的最低要求？
- P1（应该有）：哪些功能是重要但不是必须的？
- P2（可以有）：哪些功能是锦上添花的？
- **合格**：每个功能都有明确的用户触达条件
- **不合格**："需要一个搜索功能"没有说明搜索范围、排序、筛选方式

### 技术选型维度
- 前端框架有偏好吗？如无，基于项目复杂度推荐
- 后端/服务端需要的技术栈？无服务器还是有服务端？
- 是否涉及 AI/大模型能力？
- 数据存储方案？数据库类型（关系型/文档型）？

### 数据存储维度
- 需要存储哪些核心数据实体？
- 数据量级预估？
- 是否需要文件存储（图片/文档上传）？
- 是否有数据导出/备份需求？

### 导出能力维度
- 用户是否需要导出数据？（PDF/Excel/CSV）
- 导出的触发条件是什么？（手动/定期自动）
- 导出的数据范围是什么？

### 性能要求维度
- 预期用户量和并发量？
- 页面加载时间要求？
- 是否有离线使用需求？

### 界面布局维度
- 是否需要响应式设计（适配手机/平板/桌面）？
- 是否有品牌色或 VI 规范？
- 信息密度偏好：简洁型 / 功能型 / 数据密集型？

### 平台支持维度
- 需要支持哪些浏览器（兼容 IE 吗）？
- 是否需要多语言国际化？
- 是否需要多主题（浅色/深色模式）？
