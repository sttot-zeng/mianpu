TypeScript 中 type 和 interface 的区别是什么？什么情况下应该使用哪个？
简单
interface 支持声明合并，更适合定义对象结构的契约；type 可以为联合类型、交叉类型等复杂类型起别名。优先使用 interface 定义公共 API 的类型，使用 type 处理联合类型、工具类型等场景。
解释 React 中 Virtual DOM 的工作原理及其性能优势。
中等
Virtual DOM 是真实 DOM 的轻量级 JS 对象表示。当状态变化时，React 通过 Diff 算法比较新旧 VDOM 树的差异，计算出最小化的 DOM 操作，再批量更新真实 DOM。优势：减少直接 DOM 操作次数、跨平台能力、声明式编程模型。
什么是闭包？在实际项目中有哪些应用场景？
简单
闭包是函数与其词法环境的组合。应用场景：防抖/节流函数、React Hooks（useState/useEffect 依赖闭包保存状态）、模块化私有变量、柯里化、高阶组件等。
如何优化 React 应用的首次渲染速度？请列出至少三种方案。
中等
1）代码分割（React.lazy + Suspense）按需加载；2）服务端渲染（Next.js）减少白屏时间；3）关键 CSS 内联、预加载关键资源；4）图片懒加载；5）Bundle 分析优化，移除未使用的依赖。
CSS 的 BEM 命名规范是什么？有什么优缺点？
简单
BEM 即 Block__Element--Modifier 命名模式。优点：命名清晰、组件间样式隔离、可维护性强。缺点：类名较长、在深层嵌套时不够灵活。适合大型项目团队协作。
讲一下微前端架构的优缺点以及适用场景。
中等
优点：独立开发部署、技术栈无关、增量迁移、隔离性。缺点：加载性能开销、样式/事件隔离复杂、公共依赖管理。适用场景：大型企业级应用、多团队协作、遗留系统渐进式重构。

项目深挖
在前端技术栈中的落地实践 · 4 题
▼
请详细描述你最引以为豪的一个前端项目的技术架构。
困难
我主导设计的企业低代码平台前端架构采用微内核模式：核心层负责组件生命周期和渲染调度，插件层承载具体业务组件，通信层通过发布订阅模式实现跨组件协作。技术栈：React + TypeScript + Zustand + Monorepo。
在这个项目中，你做了哪些技术选型决策？背后考虑是什么？
中等
选择 Zustand 而非 Redux：API 简洁、TypeScript 支持好、不需要样板代码。选择 Monorepo（Turborepo）：共享类型定义和工具函数，统一构建流程。选择 Vite：开发启动快、HMR 体验优秀。
项目开发中遇到过最大的技术挑战是什么？如何解决的？
困难
低代码画布的性能问题：100+ 组件同时渲染导致交互卡顿。解决方案：虚拟列表（仅渲染可视区域组件）+ 增量渲染（非关键组件延迟渲染）+ 快照对比（避免不必要的重渲染），最终 60fps 流畅运行。
如果重新做这个项目，你会做哪些不同的设计决策？
中等
1）早期引入端到端测试（Playwright）而不是后期补；2）设计阶段就考虑国际化方案，避免后期大规模重构；3）组件库采用更严格的版本管理规范。

STAR 法则回答 · 4 题
▼
描述一次与产品经理在需求上的分歧，你是如何处理的？
中等
运用 STAR 法则：情境（Situation）— 产品经理要求一周内完成复杂功能开发；任务（Task）— 评估技术可行性和开发成本；行动（Action）— 提供最小可行版本方案并展示技术约束；结果（Result）— 达成共识，分两期交付。
说一个你在项目中推动技术改进的经历。
简单
主动推动组内引入 TypeScript 迁移计划。先在小模块做 POC 验证收益，然后制定渐进式迁移路线图，编写迁移指南文档，组织 Code Review 保障质量。半年内团队所有新代码均使用 TypeScript 编写。
如果在截止日期前发现需求有重大问题，你会怎么做？
中等
1）立即评估影响范围和严重程度；2）及时同步项目经理和相关方；3）提供多个备选方案（延期/降级/分阶段）；4）制定应对计划并跟进执行。关键在于早暴露、早沟通。
如何评估是否采用一个新的技术方案？请举例说明。
中等
多维度评估：技术成熟度、团队熟悉度、社区活跃度、维护成本。举例：评估 Vite vs Webpack 时，我们用新项目做两周 POC，对比构建速度（5x 提升）、HMR 体验、插件生态覆盖度，最终决定在新项目中采用 Vite。
